### 🔥 Critical Issue 2: Schema.org emailのプライバシーリスク解消

**問題点（レビュー指摘）:**
```json
"email": "dragonrondo@gmail.com"
```
- メールアドレスをHTML内にプレーンテキストで公開
- スパムボット・自動収集ツールの標的になるリスク
- 構造化データはクローラーに優先的にスキャンされる
- メールアドレス変更時にSchema.orgも更新が必要（保守コスト増）

**修正内容:**
```json
// email削除 → contactPointに変更
"contactPoint": {
  "@type": "ContactPoint",
  "contactType": "Professional",
  "url": "https://tenormusica2024.github.io/portfolio/index.html#contact",
  "availableLanguage": ["Japanese", "English"]
}
```

**効果:**
- ✅ プライバシーリスク完全解消
- ✅ スパムボット対策
- ✅ コンタクトフォームURLで連絡可能性を維持
- ✅ 言語対応情報を明記（日本語・英語）

**リスク評価（修正後）:**

- SEO上の必要性: **低**（emailなしでも問題なし）

### 🔥 Critical Issue 3: occupationalCategoryの不正確性を解消

**問題点（レビュー指摘）:**
```json
"occupationalCategory": "15-1299.09"
```
- 「15-1299.09」は米国O*NET職業分類コード
- 「AI Engineer」に完全一致する標準コードは存在しない可能性
- 不正確なコードはGoogle検索結果で誤情報として扱われるリスク
- Schema.org validatorでエラーが出る可能性

**修正内容:**
```json
// occupationalCategoryを削除（安全策）
"hasOccupation": {
  "@type": "Occupation",
  "name": "AI Engineer",
  "description": "AI駆動開発、データ分析、Webアプリケーション開発を専門とするエンジニア"
}
```

**効果:**
- ✅ 不正確なコード削除によるvalidation error回避
- ✅ nameとdescriptionで十分な情報提供
- ✅ Schema.org仕様に準拠した安全な構造化データ

**検証結果:**
- Schema.org必須プロパティ: name（✅存在）
- 推奨プロパティ: description（✅存在）
- オプショナル: occupationalCategory（削除してもOK）
